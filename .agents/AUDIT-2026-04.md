# Six Beans Coffee — Code Audit (Tier 1 + 2)

_Date: 2026-04-23 · Scope: backend security + analytics ingestion correctness_

This document ranks findings by severity. Each item has a concrete file:line reference and a fix note. Triage suggestion at the bottom.

---

## 🔴 CRITICAL

### C1. Public `/register` endpoint lets anyone create an owner account
**File:** `backend/app/routers/auth.py:153-178`

`POST /api/auth/register` has **no auth dependency and no role restriction**. Any unauthenticated client can POST `{"email": "...", "role": "owner", ...}` and become an owner of the entire system. The handler reads `user_data.role` directly from the request body (line 166) with no whitelist or admin check.

**Fix:** Either delete the endpoint entirely (users should only be created by an owner via the existing owner-only flow in `/api/users`), or lock it behind `Depends(require_roles(UserRole.owner))` and reject any role claim in the payload — force new users to start as `employee`.

### C2. Kiosk endpoints return schedule data without authentication
**File:** `backend/app/routers/kiosk.py:276-326`

`GET /api/kiosk/locations` and `GET /api/kiosk/schedule/{location_id}` have no auth dependency. Anyone on the public internet can fetch every active location ID and the daily schedule — employee names, IDs, shift times — for any shop.

**Fix:** If kiosks need an unauthenticated endpoint, gate it on a per-device token (`X-Kiosk-Token` header) tied to a specific location. Otherwise require a full authenticated session.

### C3. SystemSettings `None` crash on fresh / unseeded DB
**File:** `backend/app/routers/insights.py:683-685`

The `/insights/elite-scorecards` endpoint does `settings_row = (... SystemSettings).scalar_one_or_none()` then accesses `settings_row.labor_burden_multiplier`. If the settings row has never been inserted (fresh deploy, new customer, or post-wipe), this is `AttributeError: 'NoneType' has no attribute …` → 500 on the Insights page.

**Fix:** Reuse the existing `_settings_row(db)` helper from `backend/app/routers/expenses.py:65` which auto-creates the row on first access. Apply the same pattern everywhere `SystemSettings` is queried.

## 🟠 HIGH

### H1. Default JWT secret hardcoded in source
**File:** `backend/app/config.py:13`

`jwt_secret_key: str = "change-me-in-production"` — if a deploy forgets to set `JWT_SECRET_KEY`, the app boots with a known string and any attacker can forge tokens.

**Fix:** Remove the default, require `JWT_SECRET_KEY` as a mandatory env var, and fail to start if it's empty or matches the placeholder.

### H2. Unique constraints declared in SQLAlchemy models but may not exist in DB
**Files:**
- `backend/app/models/daily_revenue.py:38-40` — `UniqueConstraint("location_id", "date", "channel")`
- `backend/app/models/location.py:26` — `canonical_short_name unique=True`
- `backend/app/models/location.py:35-36` — `tapmango_location_id`, `doordash_store_id unique=True`

These are declared in Python but the DB only gets them at `create_all()` time. On an older DB where these columns were added later via `ALTER TABLE ADD COLUMN`, the unique index never got created — duplicates can silently accumulate, then `scalar_one_or_none()` calls crash with `MultipleResultsFound`. We've already hit this at least twice (Bakery duplicate, Victorville duplicate).

**Fix:** Add explicit `ALTER TABLE … ADD CONSTRAINT IF NOT EXISTS …` at startup for each unique constraint, following the pattern already used for the `ck_ingestion_runs_source` CHECK constraint at `main.py:145-151`.

### H3. Location deletion orphans child rows in 4 of 5 tables
**File:** `backend/app/models/location.py:48`

Only `daily_revenues` has `cascade="all, delete-orphan"`. When row 8 was deleted during the Locations cleanup, rows in `hourly_revenue`, `daily_labor`, `expenses`, and `ingestion_runs` were left orphaned — their `location_id` still points at the deleted row.

**Fix:** Either add `ondelete=CASCADE` to every FK to `locations.id`, or switch to soft delete (`is_active=False`) and filter out inactive locations in analytics queries.

### H4. Kiosk PIN brute-force has no rate limiting
**File:** `backend/app/routers/kiosk.py:82-159`

`/kiosk/authenticate` validates 4-digit `pin_last_four` against the DB with no lockout, no backoff, no CAPTCHA. ~10,000 guesses per employee is trivial to automate.

**Fix:** Add per-(location, IP) rate limit (e.g., 5 attempts / 15 min lockout) and a monotonic `failed_kiosk_attempts` counter on the user record.

### H5. Batch ingest loads location map once; concurrent edits cause misrouting
**File:** `backend/app/routers/analytics_admin.py:853-877`

The `/ingest/batch` endpoint loads `locations = … .scalars().all()` and builds `by_godaddy_id`, `by_tapmango_id`, etc. at the top. If an owner edits a mapping in another tab mid-upload, the dicts are stale — data routes to the old location_id.

**Fix:** Lock the `locations` table (`SELECT … FOR UPDATE`) at the start of the batch, or re-fetch the map before each file.

### H6. Public registration allows arbitrary role assignment (overlaps C1)
**File:** `backend/app/routers/auth.py:166`

Even if the endpoint is kept public, `user_data.role` is passed through untouched. Strip it from the payload server-side and force `role = UserRole.employee`.

## 🟡 MEDIUM

### M1. IDOR on `GET /api/locations/{id}` and `GET /api/users/{id}`
**Files:** `backend/app/routers/locations.py:29-39`, `backend/app/routers/users.py:157-178`

A manager at one shop can fetch another shop's detail or any user by guessing IDs. Role check passes, but no per-resource ACL.

**Fix:** Verify `current_user.locations` contains the requested `location_id`, or for `users`, that the requested user shares a location with the caller.

### M2. CORS config allows wildcard + credentials combo if env mis-set
**File:** `backend/app/main.py:49-50`

If `CORS_ORIGINS="*"` is ever set in Render env, `allow_credentials=True` makes authenticated cross-origin requests possible from any site. Browsers will reject it, but strict validation client-side isn't guaranteed.

**Fix:** At startup, raise if `allow_origins` contains `*` when `allow_credentials=True`.

### M3. Parsers silently coerce corrupt cells to 0
**File:** `backend/app/services/parsers/godaddy_excel.py:48-57` (and tapmango / doordash equivalents)

`_as_float()` returns `0.0` on any parse failure. A single corrupt row silently under-reports that day's revenue with no log.

**Fix:** Return a `(value, ok)` tuple, track the count of failures in `ParsedRevenueRow.raw_notes`, surface as a warning on the admin page.

### M4. GoDaddy terminal ID collisions not detected
**File:** `backend/app/routers/analytics_admin.py:870-877`

If two locations are assigned the same terminal UUID (typo in `CANONICAL_MAPPINGS`), settlement files route to whichever Location was last seen. No unique check.

**Fix:** At startup, build `set(terminal_ids)` across all Locations and log/raise if any terminal appears more than once.

### M5. `IngestionRun` FK has no `ondelete` behaviour
**File:** `backend/app/models/ingestion_run.py:70`

Deleting a Location silently leaves dangling `ingestion_runs.location_id` references. Low impact (history only) but shows up as broken joins in the admin UI.

**Fix:** `ForeignKey("locations.id", ondelete="SET NULL")`.

### M6. Startup migrations are not atomic
**File:** `backend/app/main.py:91-252`

A failure partway through the long list of `ALTER TABLE` + seed + backfill operations can leave the DB half-migrated. Next boot retries but ordering assumptions are fragile.

**Fix:** Move to Alembic proper — it's already in the deps (`alembic>=1.13.0` in `requirements.txt`) but not wired up.

### M7. `sixbeans-ingest` service in render.yaml keeps failing
**File:** `render.yaml` (cron entry)

The nightly Playwright scraper is disabled in `ALL_SOURCES` but the Render service still tries to build and shows red in the dashboard.

**Fix:** Delete the service block from `render.yaml`.
